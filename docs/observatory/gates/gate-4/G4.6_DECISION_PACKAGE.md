### 1. aiosqlite ResourceWarning

**Initial observation:** The earlier 178-test execution passed with exit code 0
but emitted a post-test aiosqlite ResourceWarning:

```
ResourceWarning: <aiosqlite.core.Connection object at ...> was deleted before
being closed.
```

**Diagnosis:** The warning was traced to `TestInitDbHardening::test_init_db_allowed_with_true_flag`
in `portal/tests/test_gate4_hardening.py`. This test called `await init_db()`,
which uses the module-level engine from `portal/database.py` (line 33). That
engine created an aiosqlite connection that was never disposed because the test
did not call `close_db()` in its cleanup path.

**Correction:**
- Primary fix: Added `await close_db()` in the `finally` block of
  `test_init_db_allowed_with_true_flag` to dispose the module-level engine.
- Defense in depth: Converted all SQLite `:memory:` fixtures in
  `test_gate4_hardening.py` (4 fixtures) and `test_observatory.py` (1 fixture)
  to file-based temp databases with `try/finally` cleanup and explicit
  `await session.close()` before `await engine.dispose()`.

**Verification:**
- SQLite-only run: 122 passed, 5 expected PostgreSQL skips, **0 warnings**
- PostgreSQL-enabled run: 127 passed, 0 skipped, **0 warnings**

### 2. PostgreSQL Migration-Backfill Failure

**Initial observation:** The PostgreSQL migration-backfill test
(`test_hash_version_backfill_postgresql`) failed with
`DuplicateTable: relation "audit_logs" already exists`.

**Diagnosis:** The `gate4_test` database had leftover tables from a different
schema (created by `Base.metadata.create_all` in concurrency tests). The
baseline migration's `CREATE TABLE` conflicted with these existing tables.

**Correction:** Rewrote `scripts/rebuild_gate4_test.py` to drop and recreate
the entire `gate4_test` database before running Alembic migrations. Also updated
the test to fully drop/recreate the disposable database rather than attempting
to clean individual tables.

**Successful rerun:** The PostgreSQL migration-backfill test passed after the
correction. Verified: v1 hashes preserved, v2 events get `hash_version=2`,
mixed v1/v2 chain verifies successfully.
