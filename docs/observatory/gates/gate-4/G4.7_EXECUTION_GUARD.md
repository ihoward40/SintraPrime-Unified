# G4.7 — Centralized Execution Guard

**Date:** 2026-07-16  
**Branch:** `feat/observatory-g4.7-execution-guard`  
**Decision:** PASS

---

## Mission

Establish one authoritative enforcement boundary for every consequential agent action.

**Constitutional rule:** No consequential action may execute unless it passes through the centralized execution guard.

The guard evaluates:
- Global kill-switch state
- Mission state
- Action classification (risk level, governance gate, approval requirement)
- Agent status
- Principal context (G4.8 placeholder)
- Requested scope and resource target
- Applicable exemptions

### Kill-Switch Matrix

- **Permitted:** kill-switch activation, evidence reads, event reads, replay, incident review, health checks, file reads, credential reads

## Guard API

### evaluate()

```python
decision = await ExecutionGuard.evaluate(
    session=db,
    action=ExecutionAction.FILE_MODIFY,
    mission_id=mission_id,
    agent_id=agent_id,
    resource_type="file",
    resource_id=path,
    principal_context=principal_context,
)
```

Returns `ExecutionGuardDecision` without raising.

### require_allowed()

```python
await ExecutionGuard.require_allowed(
    session=db,
    action=ExecutionAction.FILE_MODIFY,
    mission_id=mission_id,
)
```

Raises a typed exception on denial. Returns decision on success.

### Typed Exceptions

- `KillSwitchActiveError` — kill switch is active
- `MissionStateBlockedError` — mission state blocks the action
- `ApprovalRequiredError` — approval not obtained
- `GovernanceGateDeniedError` — governance gate check failed
- `ExecutionDeniedError` — other denial (fail-closed, unknown action)

## Service Integration Points

The guard is called at service boundaries, not only routers:
- `MissionService.create()` — calls `require_execution_allowed(ActionType.MISSION_CREATE)`
- Existing `require_execution_allowed()` backwards-compatible function delegates to new guard

Additional integration pending:
- `MissionService.update_status()` — MISSION_START, MISSION_RESUME, MISSION_CANCEL
- `AgentService.register()` — SUBAGENT_SPAWN
- Kill-switch clear path — KILL_SWITCH_CLEAR
- External communication services
- Repository publication paths

---

## Failure Semantics

- **Fail closed** for consequential actions when kill-switch state cannot be determined
- **Fail open** for read-only diagnostics (documented exception: `KillSwitchBehavior.ALLOWED` actions skip the DB lookup entirely)
- Unknown actions fail closed
- Unknown mission states fail closed
- Audit emission failures do NOT block the decision

---

## G4.8 Integration Contract

`PrincipalContext` is a placeholder:
- `subject_id`, `authentication_method`, `roles`, `permissions`, `is_authenticated`
- G4.7 does NOT trust actor names from request bodies
- G4.7 does NOT claim full authorization is complete
- Test principals can be injected via `PrincipalContext.for_testing()`
- G4.8 will replace this with authenticated principal identity

---

## Known Limitations

1. **Kill-switch clear authorization placeholder:** G4.7 requires approval for `KILL_SWITCH_CLEAR` but uses a temporary placeholder principal. G4.8 will supply authenticated principals.
2. **TOOL_INVOKE conditional approval:** The `CONDITIONAL` approval behavior is defined but the condition logic (e.g., based on tool risk classification) will be implemented in G4.8.
3. **Service integration is partial:** `MissionService.create()` is integrated. Other service methods need guard calls added in follow-up.
4. **Audit event emission can be disabled:** `_audit_enabled` class variable allows tests to suppress audit events to avoid polluting event-count assertions. This is a test-only mechanism.
