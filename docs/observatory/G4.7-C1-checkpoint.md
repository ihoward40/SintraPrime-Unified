## 3. Route-to-Policy Matrix (seven hardened routes)

| Route | Method | RBAC role (require_audited_role) | ExecutionGuard action | Kill-switch blocks? | Service mutation |
|-------|--------|----------------------------------|----------------------|--------------------|-----------------|
| `/api/v1/events` | POST | FIRM_ADMIN | EVENT_APPEND | Yes (BLOCKED) | EventService.create |
| `/api/v1/missions` | POST | FIRM_ADMIN | MISSION_CREATE | Yes (BLOCKED) | MissionService.create |
| `/api/v1/approvals` | POST | FIRM_ADMIN | APPROVAL_REQUEST | Yes (BLOCKED) | ApprovalService.create |
| `/api/v1/approvals/{id}/decide` | POST | FIRM_ADMIN | APPROVAL_DECISION | Yes (BLOCKED) | ApprovalService.decide |
| `/api/v1/evidence` | POST | FIRM_ADMIN | EVIDENCE_SUBMIT | Yes (BLOCKED) | EvidenceService.create |
| `/api/v1/emergency/kill-switch` | POST | SUPER_ADMIN | KILL_SWITCH_ACTIVATE | n/a (activates) | KillSwitchService.activate |
| `/api/v1/emergency/kill-switch` | DELETE | SUPER_ADMIN | KILL_SWITCH_CLEAR | n/a (clears) | KillSwitchService.clear |

Notes:
- `require_audited_role` emits BOUNDARY_REQUESTED → BOUNDARY_AUTHORIZED on success, and BOUNDARY_REFUSED (detail=`insufficient_role`) on role denial, before any service call.
- The ExecutionGuard is evaluated inside the route body AFTER authorization; a guard denial raises 403 with no domain mutation (G-B3).
- Kill-switch activate/clear idempotent at the service layer; clear returns `None` when no active switch.

## 4. Audited Dependency Graph

```
HTTP request
  └─ get_current_user            (real JWT decode; 401 on missing/invalid — OUT OF C1 SCOPE)
       └─ require_audited_role(required_role, action)   [C1 composite]
            ├─ _build_principal(user)        → PrincipalContext (RBAC→guard role bridge)
            ├─ _resolve_correlation_id(req)  → correlation_id (body.metadata.correlation_id or generated)
            ├─ _write_boundary_event(REQUESTED, fail_closed=True)   → commit
            ├─ assert_role_allowed(user, required_role)  [shared w/ require_role]
            │     └─ on denial: _write_boundary_event(REFUSED, insufficient_role, fail_closed=True) → commit → 403
            └─ _write_boundary_event(AUTHORIZED, fail_closed=True) → commit → returns PrincipalContext
                 └─ route body:
                      ├─ ExecutionGuard.require_allowed(action, principal_context)  [G-B3]
                      │     └─ on denial: _record_boundary_event(REFUSED, guard reason) → 403 (no mutation)
                      ├─ service mutation (flush only)
                      ├─ on success: _record_boundary_event(EXECUTED)  [best-effort]
                      └─ on exception: db.rollback(); _record_boundary_event(FAILED)  [best-effort]
```

No other router in the portal was modified. The dependency is NOT introduced globally (per instruction).

## 5. Role Translation (RBAC ↔ ExecutionGuard vocabulary)

The portal RBAC vocabulary (`SUPER_ADMIN`, `FIRM_ADMIN`, …) differs from the
ExecutionGuard policy vocabulary (`system_admin`, `incident_commander`).
`_build_principal` bridges them without granting new authorities:

| RBAC role | Guard roles added |
|----------|-------------------|
| SUPER_ADMIN | `system_admin`, `incident_commander` |
| FIRM_ADMIN | `system_admin` |
| others | (none) |

The kill-switch guard policies require `{system_admin, incident_commander}`; a
real SUPER_ADMIN therefore satisfies them. RBAC already refuses FIRM_ADMIN/lower
at the dependency layer, so the bridge never elevates a non-privileged user.

---

## 6. Guard Behavior

- `ExecutionGuard.require_allowed` is called in the route body for every
  hardened mutation (G-B3). A BLOCKED action (active kill switch) returns
  `allowed=False` → route raises 403, no domain mutation.
- KILL_SWITCH_ACTIVATE / KILL_SWITCH_CLEAR are CLEARING actions (not blocked by
  an active switch); they require an authenticated principal with the mapped
  guard roles (satisfied only by SUPER_ADMIN after the RBAC gate).
- The guard only flushes; the surrounding transaction is committed by the route.

---

## 7. Kill-Switch Rules

- Activation: SUPER_ADMIN only (RBAC). Identity (`activated_by`) derived from the
  authenticated principal — `request.activated_by` is no longer trusted (schema
  field made optional). Service is idempotent. Emits `KILL_SWITCH_ACTIVATED`.
- Clearing: SUPER_ADMIN only (RBAC). `principal_context` passed into the service
  so its internal guard enforces the same policy. `cleared_by` derived from
  principal. Emits `KILL_SWITCH_CLEARED`. Returns `None` when no active switch.
- While active: all BLOCKED actions (events, missions, approvals, evidence,
  approval decisions) are refused at the guard with 403 and no mutation.

## 8. Audit Lifecycle, Correlation, Recursion, Transactions

- **Lifecycle** (one correlation_id throughout):
  - insufficient role: REQUESTED → REFUSED (no mutation)
  - guard refusal: REQUESTED → AUTHORIZED → REFUSED (guard reason)
  - success: REQUESTED → AUTHORIZED → EXECUTED
  - service failure: REQUESTED → AUTHORIZED → FAILED
- **Correlation:** extracted from `body.metadata.correlation_id` when present,
  else `uuid4().hex` generated and stored on `Starlette Request.state` so the
  route body reuses the identical value.
- **Recursion prevention:** `_record_boundary_event` suppresses EXECUTED/FAILED
  for `action == EVENT_APPEND` (the underlying `ObservatoryEvent` IS the
  execution record). REQUESTED/AUTHORIZED/REFUSED are still emitted for appends.
- **Transaction semantics:** `EventService.create` only `flush()`es (never
  commits) — verified at `portal/services/observatory_service.py:183`. Boundary
  events therefore commit independently via `_write_boundary_event`. REQUESTED and
  REFUSED use `fail_closed=True`: if the audit write fails, the request is
  refused (500) — audit failure can NEVER grant access. EXECUTED/FAILED use
  `fail_closed=False` (best-effort log) so a transient audit problem never masks
  a real result nor rolls back a legitimate mutation. On service exception the
  route calls `await db.rollback()` before recording FAILED, so a failed mutation
  never leaks into committed state. Boundary events use a dedicated `run_id=
  "boundary"` and never disturb the domain hash chain.

## 9. Test Layer Classification (important scope boundary)

The isolated-router tests are **ISOLATED REAL-ROUTER POST-AUTH BOUNDARY TESTS**:
- They exercise the real `CurrentUser` model, real `require_role` /
  `require_audited_role` / `get_principal_context` derivation, real route
  dependency composition, real `ExecutionGuard`, real DB-backed route behavior
  (Observatory tables on SQLite), and real boundary-event persistence.
- `get_current_user` is overridden to return a real `CurrentUser` built from a
  JWT-shaped payload, so **authentication-transport behavior is NOT proved**:
  missing/invalid/expired JWT → 401, bearer parsing, and full `create_app()`
  PostgreSQL wiring are exercised by the portal's existing auth/integration
  tests, not by this file.
- SQLite is faithful for the Observatory models (no JSONB/ARRAY/PG-enum/
  UUID-server-default); `EventService.create`'s `SELECT … FOR UPDATE` is bypassed
  via a monkeypatched `skip_lock=True` (single-threaded test seam only — no
  production code altered).

### `pytest portal/tests/` exit-code-5 quirk (documented separately)

Invoking `pytest portal/tests/` (bare directory) from the repo root returns
**exit code 5 (no tests collected)**. This is a collection/rootdir configuration
quirk in this checkout (the directory is not recognized as a test root under the
current `pytest.ini`/`pyproject` settings, or import-mode prevents discovery),
**NOT a test failure** — every test file runs green when named explicitly, and
the CI/run_tests.sh wrapper discovers them correctly. Do not interpret exit 5 as
regression. Use the explicit file list above (or `scripts/run_tests.sh`) for
reproduction.
